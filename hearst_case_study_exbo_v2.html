The problem: 10 fragmented source exports, 100K+ raw records, zero shared schema
Starting state · 9 tool exports + CAQ from TCS · manual, error-prone, weeks of SAM prep per quarter
No single system held a complete picture of Hearst's application estate. Security queried Wiz, CrowdStrike, Tenable, and Zscaler. IT used JAMF/SCCM and SAMPro. Finance pulled Concur and SpendHQ. Department heads maintained ad-hoc spreadsheets. And critically — a Critical Application Questionnaire (CAQ), a structured third-party survey conducted in 2023 by TCS, provided the most structured input but covered only the applications deemed mission-critical at that point in time. The same app appeared under 3–5 different names across sources ("SFDC", "Salesforce", "salesforce.com"). No shared owner field. No BU mapping. No risk rating. Quarterly SAM reviews required weeks of manual consolidation per cycle with no repeatable process and no audit trail.

The pipeline: 9-step application intelligence architecture
Not a spreadsheet cleanup — a decision-grade data normalization and enrichment system
One-sentence summary: a multi-stage pipeline transforming raw IT telemetry into a decision-grade application intelligence model for cost optimization, risk reduction, disaster recovery planning, and SaaS governance.
Steps 1–5 · Normalization layer
Step 1 — Pre-classification: App names normalized (lowercase, .exe/.app stripped, versions removed). Libraries, runtimes, CLI tools, drivers filtered out. Every record classified as True Application / Tool / Non-App / Unknown before any analysis. Eliminated 20–30% noise upfront.

Step 2 — Canonicalization: All variants merged to one identity. "ms word" → Microsoft Word. "vs code" → Visual Studio Code. Salesforce Sandbox → Salesforce. Canonical App Name + Deduplication Key (Normalized + Vendor) created per record. Converts 100K+ rows to ~1,800–2,200 real apps.

Step 3 — Final validation: Re-ran classification on canonical names. Anything still resembling an installer, runtime, or driver reclassified before moving forward.

Step 4 — User count aggregation (critical ordering): Sum "Number of Users" if present; else count distinct Server Name. Done BEFORE deduplication — aggregating after the merge would cause undercounting for any app split across multiple source records.

Step 5 — Deduplication: Grouped by Canonical App Name + Vendor. SUM user counts; FIRST for all metadata. All 49 original source columns preserved — never dropped.
Steps 6–9 · Enrichment layer
Step 6 — Category enrichment (the bottleneck): Assigned each app to 1 of 14 business categories via multi-layered logic: substring matching (zoom/teams → Collaboration, sql/oracle → Database), vendor-based rules (Microsoft → Productivity, Adobe → Creative), pattern recognition (editor/studio → Dev/Creative), fallback AI classification in batches of 150–250, then human + AI review loop. Slow by design — solving semantic ambiguity at scale.

Step 7 — RTO/RPO tiering: Tier 0 (mission-critical, minutes recovery) through Tier 5 (non-critical, weeks). App function and business impact validated tiers beyond raw data. This made the dataset a resilience model, not just an inventory.

Step 8 — MFA/SSO tagging: Okta-connected → SSO+MFA. SSO field → SSO Enabled. MFA field → MFA Enabled. Else → Unknown. Surfaced 613 Tier 0/1 apps with Unknown auth status — the core risk finding.

Step 9 — Final export workbook: 4 worksheets: True Applications, Tools, Non-Apps, Unknowns. All 49 original columns + normalization fields + canonical names + user counts + 14-category label + RTO/RPO tier + MFA/SSO status.

The opportunity Torii's AI classification surfaced
Once the 2,300+ app inventory was live in Torii with identity system integrations pulling real user counts, Torii's AI classification engine grouped all applications by category and function — revealing that the project management category alone contained approximately 70 tools across Hearst's 11 business units, most with significant functional overlap and many with low active utilization.
The situation
Asana, Monday.com, Smartsheet, multiple Jira variants, Airtable, Basecamp, Trello, and others — all running simultaneously across different business units with no cross-BU coordination, no standardization, and compounding license costs. Different teams had independently adopted tools to solve the same problems, with no central visibility into how many tools existed in the category or how heavily each was actually used.
The approach
Used Torii's AI tagging to classify all 70 apps by sub-function. Pulled active user counts from integrated Okta and Entra ID to separate genuinely used tools from dormant licenses. Mapped functional overlap across BUs to identify which platforms covered the majority of real organizational use cases. Ran stakeholder alignment sessions with BU leads using the utilization data to build consensus around a migration path.
